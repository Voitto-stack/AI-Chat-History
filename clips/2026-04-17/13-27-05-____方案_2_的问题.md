### 方案 2 的问题

- **需要重写所有查询** — 工作量巨大
- **失去类型安全** — 没有 Prisma 的类型推导
- **SQL 手写容易出错** — 需要手动处理参数绑定

## 方案 3：混合方案（推荐）

### 核心思路

- **读操作** — 继续用 Prisma（性能好，类型安全）
- **写操作** — 强制走 DMS（审批 + 审计）

### 实现代码

```typescript
// apps/minerva-server/src/db/index.ts
import { PrismaClient } from '@prisma/client';
import { executeDMSQuery } from './dms-pool';

export class MinervaDB {
  private prisma: PrismaClient;
  
  constructor(
    private userId: string,
    private userRole: string
  ) {
    this.prisma = new PrismaClient();
  }
  
  // ========== 读操作 - 用 Prisma ==========
  
  async findUsers(where: any) {
    return this.prisma.userinfo.findMany({ where });
  }
  
  async findUserById(userId: number) {
    return this.prisma.userinfo.findUnique({
      where: { user_id: userId }
    });
  }
  
  async countUsers(where: any) {
    return this.prisma.userinfo.count({ where });
  }
  
  // ========== 写操作 - 走 DMS ==========
  
  async updateUser(userId: number, data: any) {
    const sql = `
      UPDATE userinfo 
      SET ${Object.keys(data).map((k, i) => `${k} = $${i + 2}`).join(', ')}
      WHERE user_id = $1
    `;
    const params = [userId, ...Object.values(data)];
    
    return executeDMSQuery(this.userId, this.userRole, sql, params);
  }
  
  async batchUpdateUsers(where: any, data: any) {
    const { whereClause, whereParams } = buildWhereClause(where);
    const setClause = Object.keys(data)
      .map((k, i) => `${k} = $${whereParams.length + i + 1}`)
      .join(', ');
    
    const sql = `UPDATE userinfo SET ${setClause} WHERE ${whereClause}`;
    const params = [...whereParams, ...Object.values(data)];
    
    return executeDMSQuery(this.userId, this.userRole, sql, params);
  }
  
  async deleteUser(userId: number) {
    const sql = 'DELETE FROM userinfo WHERE user_id = $1';
    return executeDMSQuery(this.userId, this.userRole, sql, [userId]);
  }
  
  async disconnect() {
    await this.prisma.$disconnect();
  }
}

// 辅助函数：构建 WHERE 子句
function buildWhereClause(where: any): { whereClause: string; whereParams: any[] } {
  const conditions: string[] = [];
  const params: any[] = [];
  let paramIndex = 1;
  
  for (const [key, value] of Object.entries(where)) {
    conditions.push(`${key} = $${paramIndex}`);
    params.push(value);
    paramIndex++;
  }
  
  return {
    whereClause: conditions.join(' AND '),
    whereParams: params
  };
}
```

### 使用方式

```typescript
// apps/minerva-server/src/routes/users.ts
import { MinervaDB } from '../db';

router.get('/users', authMiddleware, async (ctx) => {
  const { userId, role } = ctx.state.user;
  const db = new MinervaDB(userId, role);
  
  try {
    // 读操作 - 直接用 Prisma，不经过 DMS
    const users = await db.findUsers({
      regulation_status: 0
    });
    
    ctx.body = { success: true, data: users };
  } finally {
    await db.disconnect();
  }
});

router.post('/users/batch-ban', authMiddleware, async (ctx) => {
  const { userId, role } = ctx.state.user;
  const db = new MinervaDB(userId, role);
  
  try {
    // 写操作 - 走 DMS，会被拦截和审批
    await db.batchUpdateUsers(
      { regulation_status: 0 },
      { regulation_status: 2 }
    );
    
    ctx.body = { success: true };
  } catch (error) {
    if (error.needApproval) {
      ctx.body = {
        success: false,
        needApproval: true,
        approvalId: error.approvalId,
        message: error.message
      };
    } else {
      throw error;
    }
  } finally {
    await db.disconnect();
  }
});
```

### 方案 3 的优势

1. **最小改动** — 读操作无需修改，继续用 Prisma
2. **安全可控** — 写操作强制走 DMS，有审批和审计
3. **性能最优** — 读操作不经过 DMS，没有额外开销
4. **类型安全** — 读操作保留 Prisma 的类型推导

## 风险与注意事项

### 风险 1：DMS 依赖

如果 DMS 服务挂了，Minerva 的所有写操作都会失败。

**缓解措施**：
- 在 DMS 不可用时降级为直接执行（记录日志）
- 监控 DMS 服务健康状态

### 风险 2：性能影响

每个写操作都要经过 DMS 的权限校验和审批判断。

**缓解措施**：
- 权限校验结果缓存（5 分钟）
- 异步写入审计日志

### 风险 3：SQL 构建错误

手写 SQL 容易出现注入漏洞或语法错误。

**缓解措施**：
- 使用参数化查询（`$1`, `$2`）
- 编写 SQL 构建函数的单元测试
- Code Review 重点检查 SQL 构建逻辑

## 总结

推荐使用**方案 3（混合方案）**：
- 读操作继续用 Prisma（快速、类型安全）
- 写操作走 DMS（安全、可审计）
- 实施成本适中，风险可控
