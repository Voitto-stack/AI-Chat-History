这件事本质上不是“代码搬家”，而是**一次架构迁移 + 业务重建**。如果你直接从“怎么迁代码”开始，很容易走偏，最后变成一堆兼容性问题和技术债。

我给你一个更靠谱的推进路径，按你现在的背景（前端 + 刚接触服务端 + 时间紧）来设计，目标是：**可评审、可落地、风险可控**。

---

# 一、先把问题“拆干净”（否则你会越做越乱）

你现在面对的是两个系统：

## 1️⃣ 老系统（sitin-fe / minerva）

* 后端：**Koa + PostgreSQL + Prisma**
* 很可能是：

  * REST API（或部分 BFF）
  * 单体 or 半单体服务
  * 数据模型已经稳定（Prisma schema）

## 2️⃣ 新系统（sitin-next）

你现在最大的问题是：👉 **不了解它的架构**

👉 这一步不能跳，否则你后面所有方案都是拍脑袋

你需要优先搞清：

* 是不是 **Next.js（App Router）+ Server Actions / Route Handlers**
* 有没有 **独立后端（Node / Nest / Go）**
* 数据层：

  * 还是 PostgreSQL？
  * ORM 还是直接 SQL？
* 是否有：

  * BFF 层（很关键）
  * 微服务拆分
  * RPC（tRPC / GraphQL）

# 三、迁移策略（不要只想一种）

你至少要给出 3 种方案（这是评审标准）

---

## ✅ 方案一：整体迁移（最干净）

👉 把 minerva-server 直接“搬进” sitin-next

做法：

* Koa → 改成 Next API Route / Route Handler
* Prisma → 复用（如果 next 也支持）
* DB → 直接复用

优点：

* 风险最低（逻辑不变）
* 开发快

缺点：

* 可能不符合 next 架构（技术债）

---

## ✅ 方案二：BFF 重写（推荐）

👉 保留数据库 + 重写服务层

做法：

* DB（PG + Prisma）保留
* Service 层迁移
* API 用 Next 重新实现（BFF）

优点：

* 架构更统一
* 适配 next

缺点：

* 工作量中等
* 需要理解业务

---

## ⚠️ 方案三：服务拆分（进阶）

👉 把 minerva 变成独立服务

```
sitin-next（前端+BFF）
        ↓
minerva-service（独立服务）
        ↓
DB
```

优点：

* 解耦
* 可扩展

缺点：

* 成本高（你现在不适合）

# 四、你这个业务的“坑点”（提前说清）

结合你发的流程（BFF奖励链路），迁移时会踩这些坑：

---

## 1️⃣ 状态机逻辑

你的流程本质是：

👉 用户生命周期状态机（W1 ~ W5）

如果你现在代码是：

* if else 写死 👉 ❌
* 没有状态表 👉 ❌

👉 迁移时建议：

* 抽象成 **状态机 or 枚举驱动**

---

## 2️⃣ 奖励一致性（非常关键）

涉及：

* 金额发放
* 提现
* 多次触发

👉 必须保证：

* 幂等性（idempotency）
* 事务（Prisma transaction）

否则迁移后一定出 bug

---

## 3️⃣ 异步 / 通知

流程里有：

* push
* polling
* 动效触发

👉 你要搞清：

* 是不是消息队列？
* 还是纯接口轮询？
