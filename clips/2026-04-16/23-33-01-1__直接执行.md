 1. 直接执行

  好，我直接把 DMS 内部拦截链 按调用顺序画出来。

  ---
  场景 1：用户在 DMS 里点“执行 SQL”

  前端起点是：

  - packages/app-dms-web/src/api/sql.ts:24
  executeSQL(payload) -> POST /sql/execute

  后端接到后，进入：

  - packages/app-dms-server/src/sql/sql.controller.ts:5
  app.post("/sql/execute", ..., postSqlExecute)

  然后进入真正的 DMS 执行入口：

  - packages/app-dms-server/src/sql/sql.service.ts:50
  postSqlExecute(req, reply)

  它整理参数后，会继续调用：

  - packages/app-dms-server/src/sql/sql.service.ts:69
  sqlExecute({ ... })

  也就是说：

  DMS 内部真正的裁决点，是从 sqlExecute(...) 开始的。

  ---
  场景 2：DMS 内部开始裁决这条 SQL 能不能直通

  位置：

  - packages/app-dms-server/src/sql/sql.service.ts:108

  这是第一道核心拦截。

  第一步：识别 SQL 类型

  - packages/app-dms-server/src/sql/sql.service.ts:109
  const category = sqlParse(sql)

  而 sqlParse 在：

  - packages/app-dms-server/src/sql/sql.service.ts:22

  它会把 SQL 分成：

  - select
  - dml
  - ddl

  所以这一步不是执行，而是：

  先把 SQL 归类。

  ---
  场景 3：如果是 DDL，DMS 直接拦下

  - packages/app-dms-server/src/sql/sql.service.ts:110

  if (category === "ddl") {
    throw new Error("DDL statements must go through the approval workflow");
  }

  这意味着像：

  - CREATE TABLE
  - ALTER TABLE
  - DROP TABLE

  这种语句，在 DMS 内部不会直达数据库。

  它在这里就被硬拦住了。

  ---
  场景 4：如果是 DML，DMS 默认也拦下

  - packages/app-dms-server/src/sql/sql.service.ts:113

  if (category === "dml" && !allowDml) {
    throw new Error("DML statements must go through the approval workflow");
  }

  也就是说：

  - INSERT
  - UPDATE
  - DELETE

  默认也不会直接执行。

  只有 allowDml === true 才会往下走。

  而 allowDml 是前面算出来的：

  - packages/app-dms-server/src/sql/sql.service.ts:67

  const allowDml = dsName.includes(TEST_DS_PATTERN) && DML_DIRECT_ROLES.has(role);

  这就表示：

  DMS 内部策略是：只有测试库 + 允许角色，DML 才能直通；否则一律拦下。

  ---
  场景 5：如果被放行，DMS 才会真正去拿数据库连接

  放行后，才进入统一连接层：

  - packages/app-dms-server/src/sql/sql.service.ts:117
  - packages/app-dms-server/src/sql/sql.service.ts:121

  逻辑是：

  const pool = database
    ? await getPoolForDatabase(dataSourceId, database, config.encryptionKey)
    : await (...) => getPool(ds)

  也就是说顺序一定是：

  1. 先经过 DMS 内部拦截策略
  2. 再走 DMS 统一数据源 / 连接池调度层
  3. 最后才可能执行 SQL

  真正执行落在：

  - packages/app-dms-server/src/sql/sql.service.ts:138
  pool.execute(dataSql)

  所以你要抓“DMS 拦截发生在哪”，核心就盯这 3 段：

  - sqlParse(sql)
  - if (category === "ddl")
  - if (category === "dml" && !allowDml)

  ---
  场景 6：被拦下来的 SQL 在 DMS 里怎么继续流转

  如果是 DML / DDL，被拦下以后，不是简单结束，而是应该转审批流。

  审批提交入口在：

  - packages/business-dms-approval/src/approvals.service.ts:33
  approvalCreate(req, reply)

  它会把这条 SQL 变成 approval record：

  - packages/business-dms-approval/src/approvals.service.ts:38
  insertApproval(dataSourceId, database ?? null, sql, req.user.id)

  这说明：

  DMS 内部的拦截不是纯 reject，而是把写操作导入另一个受控状态机。

  ---
  场景 7：审批通过后，DMS 内部还有第二道执行拦截

  真正执行审批单，在：

  - packages/business-dms-approval/src/approvals.service.ts:84
  approvalExecute(encryptionKey)

  内部真正干活的是：

  - packages/business-dms-approval/src/approvals.service.ts:106
  doExecuteApproval(...)

  这里先做状态检查：

  - packages/business-dms-approval/src/approvals.service.ts:108

  if (!approval || !["approved", "execute_failed"].includes(approval["status"] as string)) {
    throw new Error("Approval not found or not approved");
  }

  所以即使已经进了审批执行接口，也不是直接打库。

  只有审批状态已经到 approved，DMS 内部才允许继续。

  ---
  场景 8：审批链里执行前还要再过统一连接层

  通过审批状态机后，DMS 仍然不会绕开统一连接池。

  它继续走：

  - packages/business-dms-approval/src/approvals.service.ts:118
  getPoolForDatabase(...)
  - packages/business-dms-approval/src/approvals.service.ts:123
  getPool(...)

  最后才：

  - packages/business-dms-approval/src/approvals.service.ts:127
  pool.execute(approval["sql_text"] as string)

  所以审批执行链也复用了同一个 DMS 数据库调度层。

  ---
  一张最直观的 DMS 内部拦截链

  你可以直接记成这个：

  前端 executeSQL()
    -> POST /sql/execute
    -> postSqlExecute()
    -> sqlExecute()
    -> sqlParse(sql)
       -> SELECT  => 可继续
       -> DML     => 默认拦截，除非测试库 + 允许角色
       -> DDL     => 直接拦截

  如果被拦：
    -> approvalCreate()
    -> 生成 approval record
    -> approvalApprove()/approvalReject()
    -> approvalExecute()
    -> doExecuteApproval()
    -> 检查状态必须是 approved
    -> getPoolForDatabase()/getPool()
    -> pool.execute()

  ---
  2. 深度交互

  如果按第一性原理看，这条链最重要的不是“审批”两个字，而是 DMS 在内部做了一个很关键的结构分离：

  - 读操作：尽量允许直通
  - 写操作 / 结构变更：默认不直通

  所以它的本质不是“SQL 编辑器 + 审批页面”，而是：

  一个带内部裁决器的数据库访问系统。

  真正的 DMS 味道不在 UI，而在这条分流逻辑：

  - 先把 SQL 当成一种“操作意图”去判断
  - 再决定它是立即执行，还是进入审批状态机

  也就是说，DMS 的核心不是执行 SQL，
  而是：

  决定 SQL 该以什么治理路径被执行。
