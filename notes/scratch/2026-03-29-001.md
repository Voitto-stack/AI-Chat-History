# MetaBot 架构与多模型协作实现探索

## 核心发现

### 1. MetaBot HTTP API - 跨 Bot 通信模式

**文件路径**: `/Users/presence79/.claude/skills/metabot/SKILL.md`

#### 通信机制
- **底层**: HTTP REST API (默认端口 9100)
- **认证**: Bearer Token (`$METABOT_API_SECRET`)
- **核心端点**: `/api/talk` - Agent-to-Agent 跨实例通信

#### 关键通信模式
```
1. 直接通信: mb talk <botName> <chatId> <prompt>
2. 跨实例路由: mb talk <peerName>/<botName> <chatId> <prompt>
3. 自动发现: MetaBot 自动路由到正确的对等实例
```

#### Bot 与 Peer 概念
- **Local Bots**: 同一实例的 Bot
- **Peer Bots**: 其他 Claude Code 实例上的 Bot
- 通过 `mb bots` 列出所有可用 Bot（包括 peer 上的）
- 通过 `mb peers` 查看 Peer 实例状态和 URL

### 2. 任务调度与管理 - Cron + 延迟执行

**支持两种调度模式**:

#### A. 一次性任务 (Delayed)
```bash
mb schedule add <bot> <chatId> <delaySeconds> <prompt>
mb schedule cancel <id>
mb schedule list
```

#### B. 循环任务 (Cron)
```bash
mb schedule cron <bot> <chatId> '<cronExpr>' <prompt>
mb schedule pause <id>
mb schedule resume <id>
```

**时区支持**: 默认 Asia/Shanghai（可自定义）

**Cron 格式**: 标准 5 字段 (minute hour day month weekday)

### 3. 消息传递与结果收集

**工作流模式**:
```
1. Prompt 传入 → /api/talk 端点
2. Bot 执行 Claude AI 推理
3. 结果返回给调用者
4. 支持 sendCards 选项用于富文本渲染
```

**特点**:
- 同步等待结果（调用者阻塞）
- 支持 Feishu 卡片格式（`sendCards: true`）
- 支持在 Feishu/Telegram 等平台创建 Bot
- 自动化 Bot 实例管理（通过 PM2 file-watch）

### 4. Bot 创建与生命周期管理

**创建 API**:
```bash
POST /api/bots
{
  "platform": "feishu|telegram|...",
  "name": "<bot-name>",
  "feishuAppId": "...",
  "feishuAppSecret": "...",
  "defaultWorkingDirectory": "/path",
  "installSkills": true  # 自动安装技能库
}
```

**自动激活**: ~3 秒内通过 PM2 file-watch 激活
**删除**: `DELETE /api/bots/<name>`

## 多模型协作架构设计

### 核心设计哲学（来自 MetaSkill）

**Path: `/Users/presence79/.claude/skills/metaskill/flows/team.md`**

#### 1. 分布式代理队伍结构

```
主 Claude Session (CLAUDE.md 中的 Orchestrator)
        ↓
  ┌─────┼─────┐
  ↓     ↓     ↓
Specialist-1  Specialist-2  Code-Reviewer
(Sonnet)      (Sonnet)      (Sonnet)
```

**关键原则**:
- YOU 是 Orchestrator（不需要单独的 Tech Lead Agent）
- Specialist 都用 Sonnet 模型（成本效益最优）
- Code Reviewer 是质量门槛（必需）
- Max 2 个 Agent 并行执行复杂任务（避免冲突）

#### 2. Agent 路由表设计

在 `CLAUDE.md` 中定义路由表：
```markdown
| Task Type | Agent | When to Use |
|-----------|-------|-------------|
| API 设计 | api-designer | 接口设计、RESTful 审查 |
| 代码审查 | code-reviewer | 所有代码变更提交前 |
| 调试 | debugger | 错误诊断、测试失败 |
```

#### 3. 任务委派与接力模式

```
You (分析) → [Specialist] (实现) → code-reviewer (质量关卡) → 完成
```

**接力格式**:
- (a) 明确的目标
- (b) 相关文件路径
- (c) 验收标准
- (d) 下一个接力的 Agent

#### 4. 跨 Agent 知识共享 - MetaMemory

**系统**: `mm` CLI (已预装)

```bash
mm search <query>              # 搜索文档
mm create "标题" --tags "dev"  # 创建知识
mm update <doc_id> -c "..."    # 更新
mm folders                     # 浏览目录
```

**使用场景**:
- 开始工作前: 搜索已有上下文
- 完成工作后: 保存重要决策、架构笔记、发现的模式
- 发现模式时: 文档化可复用的模式供其他 Agent 参考

## 自我进化机制 - Self-Improving Agent

**Path**: `/Users/presence79/.agents/skills/self-improving-agent/`

### 多层次记忆架构

```
memory/
├── semantic/           # 抽象规则 + 最佳实践
├── episodic/           # 具体事件 + 经验
└── working/            # 当前会话上下文
```

#### 1. Semantic Memory (规则和模式)

```json
{
  "pattern_id": {
    "id": "pat-2025-01-11-001",
    "name": "文档分离模式",
    "source": "user_feedback|implementation_review|retrospective",
    "confidence": 0.95,
    "applications": 5,  # 被应用 5 次
    "category": "prd_structure",
    "pattern": "对于非平凡的 PRD，分离为 4 个文件",
    "problem": "问题描述",
    "solution": { ... },
    "quality_rules": [ ... ],
    "target_skills": ["prd-planner", "architecting-solutions"]
  }
}
```

**特点**:
- 置信度随应用次数增长
- 低置信度 (<0.5) + 30 天未用的模式自动删除
- 每个模式指向目标 Skill，自动更新相关技能

#### 2. Episodic Memory (经验和事件)

```json
{
  "id": "ep-2025-01-11-001",
  "timestamp": "...",
  "skill": "debugger",
  "situation": "用户报告数据刷新失败",
  "root_cause": "回调函数为空",
  "solution": "在回调中实现实际逻辑",
  "lesson": "始终验证回调不为空函数",
  "related_pattern": "callback_verification",
  "user_feedback": { "rating": 8, "comments": "..." }
}
```

#### 3. Working Memory (会话上下文)

- `current_session.json` - 活跃会话数据
- `last_error.json` - 错误上下文
- `session_end.json` - 会话结束标记

### 自我进化流程（4 个阶段）

```
技能完成 → 经验提取 → 模式抽象 → 技能更新 → 记忆整合
```

#### Phase 1: 经验提取
```yaml
What happened:
  skill_used: debugger
  outcome: success|partial|failure
Key Insights:
  what_went_well: [...]
  what_went_wrong: [...]
User Feedback:
  rating: 8/10
  comments: "..."
```

#### Phase 2: 模式抽象
```
实例 3+ 次 → 标记为 critical → 加入技能的"常见错误"部分
有效解决 → 标记为 best_practice → 加入"最佳实践"
用户评分 <= 4 → 标记为 weakness → 加入"避免清单"
```

#### Phase 3: 技能更新
在 Skill.md 中添加演进标记：
```markdown
<!-- Evolution: 2025-01-12 | source: ep-2025-01-12-001 | skill: debugger -->

## 模式已添加 (2025-01-12)
**Pattern**: 始终验证回调不为空函数
**Source**: Episode ep-2025-01-12-001
**Confidence**: 0.95
```

#### Phase 4: 记忆整合
- 更新 semantic-patterns.json
- 存储 episodic memory (YYYY-MM-DD-{skill}.json)
- 更新模式置信度
- 清理过时模式（低置信度 + 长期未用）

### 自我纠正 (on_error hook)

触发条件:
- Bash 返回非零退出码
- 测试失败
- 用户报告指导产生了错误结果

纠正过程:
1. **检测** - 捕获错误上下文
2. **验证** - 确认技能指导是否不正确
3. **应用** - 更新技能文件 + 添加纠正标记
4. **验证** - 测试纠正后的指导

### Hooks 集成

三个核心 Hooks:
- `pre-tool.sh` - Tool 使用前的日志
- `post-bash.sh` - Bash 执行后的错误捕获
- `session-end.sh` - 会话结束时的学习同步

## Agent 创建架构 (MetaSkill)

**设计方法 - 6 步建筑师流程**:

### Step 1: 提取核心意图
- 基本目的
- 关键责任
- 成功标准
- 隐含需求

### Step 2: 设计专家人物角色
```
"You are a senior security engineer specializing in..."
```

### Step 3: 架构全面指导
- 行为边界
- 具体方法论
- 边界情况处理
- 输出格式定义
- 项目标准对齐

### Step 4: 性能优化
- 不确定情况下的决策框架
- 质量控制机制（自验证步骤）
- 高效工作流
- 备选策略

### Step 5: 工作流纪律 (所有 Agent 必需)
```
Planning → Autonomous Execution → Verification → Self-Improvement
```

### Step 6: 创建标识符
- 小写字母、数字、连字符
- 避免 "helper", "assistant", "manager"
- 通常 2-4 个词用连字符连接

## 实现要点总结

### 消息传递
- **同步 RPC 模式**: HTTP /api/talk
- **异步调度**: Cron + 延迟执行
- **富文本支持**: Feishu 卡片格式

### 任务管理
- **中央调度**: MetaBot 管理所有任务
- **时区感知**: 支持多时区 Cron
- **暂停/恢复**: 精细控制循环任务

### 知识共享
- **三层记忆**: Semantic + Episodic + Working
- **自动清理**: 低置信度模式 30 天自动删除
- **演进标记**: 追踪每个模式的来源和版本

### 可扩展性
- **Peer 路由**: 自动发现和路由跨实例 Bot
- **模块化 Skills**: 每个 Skill 独立、可重用
- **Plugin 架构**: 通过 MCP 服务扩展功能

### 质量保障
- **Code Reviewer 门槛**: 所有代码变更必须通过审查
- **工作流纪律**: Planning → Execution → Verification
- **自纠正机制**: 错误自动触发修复和学习
