# DMS代理层排查

### User (11:22:51)
找出这里的操作数据库是否有接入一层自己的数据库代理 / 调度层，统一接管外部数据库访问，dms是我们自定义的一套数据库代理系统

### Assistant (11:22:51)
结论：

- Minerva 当前这套新包里，没有看到“独立的数据库代理 / 调度服务”。它自己的业务库访问是应用进程内直连 PostgreSQL：`app-minerva-server` 启动时直接 `createPgDb(config.dbUrl, ...)`，`config.dbUrl` 来自 `DATABASE_URL`，见 `packages/app-minerva-server/src/index.ts:13` 和 `packages/app-minerva-server/src/config.ts:9`。
- `business-minerva-auth` 访问库时也是直接走 `getPgDb()`，没有再经过单独的代理层，见 `packages/business-minerva-auth/src/auth.model.ts:17`。
- 但在老的 DMS 体系里，**对外部数据库访问确实被统一收口到你们自己的一层“应用内代理层”**：数据源管理、凭据解密、连接池创建、SQL 分类、审批、审计，都先经过 DMS server，再由它去连外部 MySQL/Postgres。
- 不过这层更准确地说是 **in-process gateway / broker**，不是独立部署的 DB proxy，也不是带调度队列的 worker 系统。

关键证据：

1. `server-plugin-pg` 自己把两种职责写得很明确：
   - `Target DB pool (used by datasources)`：给外部目标库建连接池，见 `packages/server-plugin-pg/src/index.ts:2` 和 `packages/server-plugin-pg/src/index.ts:26`
   - `App DB (used by DMS itself)`：给 DMS 自己的库建连接，见 `packages/server-plugin-pg/src/index.ts:52` 和 `packages/server-plugin-pg/src/index.ts:125`

2. DMS 的外部库访问统一从 `business-dms-datasource` 收口：
   - 维护 `pools` 映射，按 `dataSourceId[:database]` 缓存连接池，见 `packages/business-dms-datasource/src/datasources.service.ts:23` 和 `packages/business-dms-datasource/src/datasources.service.ts:42`
   - `getPool()` 根据数据源类型创建 MySQL / Postgres 连接池，见 `packages/business-dms-datasource/src/datasources.service.ts:46`
   - `getPoolForDatabase()` 先查 DMS 自己库里的数据源记录、解密密码，再返回目标库连接池，见 `packages/business-dms-datasource/src/datasources.service.ts:87` 和 `packages/business-dms-datasource/src/datasources.service.ts:107`

3. 真正执行外部 SQL 的地方，不是客户端直连，也不是独立代理进程，而是 API 请求里同步执行：
   - `/sql/execute` 路由进 `postSqlExecute`，见 `packages/app-dms-server/src/sql/sql.controller.ts:5`
   - `postSqlExecute` -> `sqlExecute` -> `getPoolForDatabase()` / `getPool()` -> `pool.execute(sql)`，见 `packages/app-dms-server/src/sql/sql.service.ts:50`、`packages/app-dms-server/src/sql/sql.service.ts:107`、`packages/app-dms-server/src/sql/sql.service.ts:116`、`packages/app-dms-server/src/sql/sql.service.ts:137`
   - 审批执行也是同一路径：`approvalExecute` 里直接取 pool 然后 `pool.execute(...)`，见 `packages/business-dms-approval/src/approvals.service.ts:83`、`packages/business-dms-approval/src/approvals.service.ts:105`、`packages/business-dms-approval/src/approvals.service.ts:117`、`packages/business-dms-approval/src/approvals.service.ts:126`

所以准确表述应该是：

- **有统一接管外部数据库访问的一层**，但它是 DMS 服务进程内部的一层数据源/连接池/执行网关。
- **没有看到独立的数据库代理服务、调度中心、队列消费者、异步 worker** 这类单独一层。
- 也就是说，现在更像“应用层数据库访问中台”，不是“独立 DB Proxy 产品”。

更进一步说，这套 DMS 已经具备你说的“自定义数据库代理系统”的几个核心特征：

- 客户端不直连目标库，只能打 DMS API
- 目标库凭据由 DMS 托管和解密
- 连接池由 DMS 统一维护
- DML/DDL 先经过分类、审批、审计，再决定是否执行，见 `packages/app-dms-server/src/sql/sql.service.ts:107`

但它还没到那种真正独立代理层的形态，比如：

- 单独部署的 proxy / worker 进程
- 统一 SQL 转发协议
- 调度队列 / 重试系统
- 连接复用与路由和 API 服务彻底解耦
