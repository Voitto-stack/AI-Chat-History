### 1. 直接执行

那就只看 DMS 这条线。我把它压成一句话：

**`app-dms-web` 负责登录和操作界面，`app-dms-server` 负责鉴权、权限控制和路由分发，真正的数据规则落在 `business-dms-*` 包里。**

按运行顺序看，后端入口在 `packages/app-dms-server/src/index.ts:0`。
启动时它先做三件事：
1. 初始化 Redis（如果配置了）
2. 建立 DMS 自己的 Postgres 连接
3. 跑迁移
然后才 `buildApp()` 并监听端口：`packages/app-dms-server/src/index.ts:7`

`buildApp()` 在 `packages/app-dms-server/src/app.ts:15`，这里就是整个 DMS 后端装配中心。它做的事很集中：
- 创建 Fastify 实例
- 注册 CORS 和 metrics
- 给每个请求挂 `preHandler` 权限钩子 `authHook`
- 注册各个 controller

也就是说，**每个请求都会先过统一鉴权/鉴权，再进业务路由。** 这一点在 `packages/app-dms-server/src/app.ts:24` 很关键。

权限判断逻辑在 `packages/app-dms-server/src/auth/auth.permissions.ts:8`。
这里不是分散写在每个 controller 里，而是维护了一张统一的 `PERMISSIONS` 表，按 `"METHOD 路径"` 映射：
- `/auth/login` 公开
- `/users/*` 只有 admin
- `/datasources/*` 登录后可读，admin 才能改
- `/sql/execute` 所有角色都能进，但后面业务层还会继续限制
- `/approvals/*` 只有 maintainer/admin 才能审批执行

所以这个系统的权限其实分两层：

第一层是 **路由层权限**，决定你能不能碰这个 API。
第二层是 **业务层权限**，决定你进来后能做多大动作。

最典型的是 SQL 执行。
路由只负责把 `/sql/execute` 挂到 `postSqlExecute`：`packages/app-dms-server/src/sql/sql.controller.ts:3`
真正规则在 `packages/app-dms-server/src/sql/sql.service.ts:43`：
- 先校验参数
- 从 `req.user` 拿当前用户角色
- 查数据源名称
- developer/viewer 只能访问名字包含 `-test` 的数据源：`packages/app-dms-server/src/sql/sql.service.ts:56`
- SQL 先用 `node-sql-parser` 分类成 `select` 或 `dml`：`packages/app-dms-server/src/sql/sql.service.ts:20`
- 非 SELECT 默认不允许直接执行，除非命中测试库 + 允许直执行 DML 的角色条件：`packages/app-dms-server/src/sql/sql.service.ts:100`
- 执行后写审计日志：`packages/app-dms-server/src/sql/sql.service.ts:140`

这就说明 DMS 的核心不是“把 SQL 发到数据库”，而是：
**把 SQL 执行包装进一套受控流程里。**

其他 controller 也是同样思路：
- `users.controller.ts` 只是把 `/users/*` 路由转发给 `@heyhru/business-dms-user`：`packages/app-dms-server/src/users/users.controller.ts:1`
- `datasources.controller.ts` 只挂数据源 CRUD：`packages/app-dms-server/src/datasources/datasources.controller.ts:9`
- `approvals.controller.ts` 负责审批流入口：`packages/app-dms-server/src/approvals/approvals.controller.ts:10`
- `audit.controller.ts` 只暴露日志查询：`packages/app-dms-server/src/audit/audit.controller.ts:3`
- `saved-sqls.controller.ts` 负责已保存 SQL：`packages/app-dms-server/src/saved-sqls/saved-sqls.controller.ts:8`

所以后端的设计很明显：
**app 层负责装路由和中间件，business 层负责真正业务。**

前端这边，入口很轻，在 `packages/app-dms-web/src/main.tsx:0`，就是 React + BrowserRouter 挂起来。
真正的页面路由在 `packages/app-dms-web/src/App.tsx:12`：
- `/login` 登录页
- `/` SQL 编辑器
- `/approvals` 审批页
- `/logs` 审计日志
- `/datasources` 数据源管理
- `/users` 用户管理

外层包了一层 `ProtectedRoute`：`packages/app-dms-web/src/App.tsx:16`
这个组件会在首次进入时调用 `/auth/me` 检查当前 token 对应的用户：`packages/app-dms-web/src/components/ProtectedRoute.tsx:18`
如果失败，就跳回 `/login`：`packages/app-dms-web/src/components/ProtectedRoute.tsx:37`

登录流程很直白：
- 登录页提交用户名密码：`packages/app-dms-web/src/pages/Login/index.tsx:12`
- 调 `login()`：`packages/app-dms-web/src/api/auth.ts:7`
- `login()` 调后端 `/auth/login`，拿到 token 后写进 localStorage：`packages/app-dms-web/src/api/client.ts:3`
- 然后把用户信息塞进 Zustand 的 `auth` store：`packages/app-dms-web/src/store/auth.ts:17`

所以前端鉴权模型不是 cookie session，而是 **Bearer token + localStorage + 启动时调用 /auth/me 回填用户态**。

前端请求统一从 `packages/app-dms-web/src/api/client.ts:17` 走。
这个 client 做了三件关键事：
- 从 localStorage 取 token 自动带上
- 401 时清 token 并跳 `/login`
- 统一弹错误消息

这意味着页面层基本不处理底层鉴权细节，API client 已经兜住了。

SQL 编辑器页是 DMS 的主页面，在 `packages/app-dms-web/src/pages/SqlEditor/index.tsx:62`。
它的工作流是：
1. 页面加载先调 `/datasources/list` 拉数据源：`packages/app-dms-web/src/pages/SqlEditor/index.tsx:20`
2. 选中数据源后调 `/sql/databases` 拉数据库列表：`packages/app-dms-web/src/pages/SqlEditor/index.tsx:30`
3. 再选数据库后调 `/sql/tables` 拉表名：`packages/app-dms-web/src/pages/SqlEditor/index.tsx:40`
4. 执行 SQL 时调 `/sql/execute`：`packages/app-dms-web/src/api/sql.ts:23`
5. 数据源、数据库选择和页面状态放在 Zustand 的 `sqlEditor` store：`packages/app-dms-web/src/store/sqlEditor.ts:14`

布局组件 `AppLayout` 还会根据角色动态裁菜单：`packages/app-dms-web/src/components/Layout/index.tsx:46`
也就是说，前端不仅是“显示页面”，它还把权限模型投影成了 UI：admin 才能看用户和数据源，viewer 只能进 SQL 编辑器。

所以你把 DMS 整条链路串起来，可以这样理解：

**用户在前端登录 -> token 存本地 -> ProtectedRoute 用 /auth/me 恢复登录态 -> 页面按角色显示菜单 -> SQL 编辑器请求数据源/数据库/表 -> 发 SQL 到后端 -> 后端先过统一 authHook -> 再进 sql.service 做更细业务限制 -> 真正调用 datasource/business 包执行 -> 最后写审计或走审批流。**

这就是 DMS 的主骨架。

### 2. 深度交互

如果从“看懂系统”这个目标出发，最关键的不是把所有页面都记住，而是先抓住这个项目的控制点。

DMS 真正的控制点其实只有 4 个：
1. `src/index.ts` / `buildApp()` —— 系统怎么启动
2. `auth.permissions.ts` —— 哪些路由谁能进
3. `sql.service.ts` —— 最敏感的 SQL 执行规则
4. 前端 `ProtectedRoute + api/client + SqlEditorPage` —— 用户怎么形成完整操作链路

也就是说，这个系统的复杂度不在“页面多”，而在“权限和 SQL 规则收得比较紧”。

如果你后面要继续读这套 DMS，最优路径不是按页面继续扫，而是往下钻这两个问题：
- `business-dms-datasource` 到底怎么封装数据库连接池
- `business-dms-approval` 到底怎么把 DML/DDL 审批流落库、审批、再执行

因为这两个地方才是 DMS 和普通后台管理系统拉开差距的地方。
