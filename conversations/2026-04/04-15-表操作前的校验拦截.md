### Assistant (23:12:53)
### 1. 直接执行

有，而且不止一层。

如果你问“在真正碰到数据表之前，会不会先被拦几次”，答案是：**会，至少 4 层。**

#### 第一层：统一登录鉴权拦截
所有请求先进 `authHook`：
- `packages/app-dms-server/src/app.ts:24`
- `packages/app-dms-server/src/auth/auth.permissions.ts:54`

这里会做：
- 看当前路由是不是要求登录：`packages/app-dms-server/src/auth/auth.permissions.ts:56`
- 从 `Authorization: Bearer ...` 里取 token：`packages/app-dms-server/src/auth/auth.permissions.ts:58`
- 没 token 直接 401：`packages/app-dms-server/src/auth/auth.permissions.ts:60`
- token 非法直接 401：`packages/app-dms-server/src/auth/auth.permissions.ts:63`
- 角色不匹配直接 403：`packages/app-dms-server/src/auth/auth.permissions.ts:68`

所以第一道闸门是：**你是不是合法用户、你这个角色能不能碰这个接口。**

#### 第二层：接口级权限拦截
就算通过登录校验，接口本身还有权限表。
和表操作最相关的是：
- `POST /sql/execute`：`packages/app-dms-server/src/auth/auth.permissions.ts:32`
- `POST /sql/tables`：`packages/app-dms-server/src/auth/auth.permissions.ts:34`
- `POST /approvals/create`：`packages/app-dms-server/src/auth/auth.permissions.ts:38`
- `POST /approvals/execute`：`packages/app-dms-server/src/auth/auth.permissions.ts:42`

这里的意思是：
- 表列表浏览，登录后才行
- SQL 执行，虽然 admin/maintainer/developer/viewer 都能进接口，但这只是“允许进入执行逻辑”，不是“允许随便改表”
- 审批执行只有 admin / maintainer 才能做

#### 第三层：SQL 业务级校验和拦截
这个才是真正和“碰表”最相关的一层。
核心在：
- `packages/app-dms-server/src/sql/sql.service.ts:50`
- `packages/app-dms-server/src/sql/sql.service.ts:108`

这里会做几类拦截：

##### 1. 参数校验
没有 `dataSourceId` 或 `sql`，直接 400：
- `packages/app-dms-server/src/sql/sql.service.ts:53`

##### 2. 数据源存在性校验
如果 data source 不存在，直接 404：
- `packages/app-dms-server/src/sql/sql.service.ts:62`
- `packages/app-dms-server/src/sql/sql.service.ts:123`

##### 3. 按角色限制能碰哪些库
`developer` / `viewer` 只能碰名字包含 `-test` 的数据源：
- `packages/app-dms-server/src/sql/sql.service.ts:63`

这意味着即使接口能进，也不代表你能碰生产库。

##### 4. SQL 类型识别
执行前先用 `sqlParse(sql)` 分类：
- `packages/app-dms-server/src/sql/sql.service.ts:22`

它会把 SQL 分成：
- `select`
- `dml`
- `ddl`

##### 5. DDL 直接拦截
如果是 `CREATE / ALTER / DROP` 这类 DDL，直接不让执行：
- `packages/app-dms-server/src/sql/sql.service.ts:110`

报错就是：
- `DDL statements must go through the approval workflow`
  `packages/app-dms-server/src/sql/sql.service.ts:111`

##### 6. DML 默认拦截
如果是 `INSERT / UPDATE / DELETE` 这类 DML，默认也不让直执行：
- `packages/app-dms-server/src/sql/sql.service.ts:113`

报错：
- `DML statements must go through the approval workflow`
  `packages/app-dms-server/src/sql/sql.service.ts:114`

##### 7. 只有受限条件下才允许 DML 直执行
只有数据源名包含 `-test` 且角色命中允许集合时，才会放开 DML：
- `packages/app-dms-server/src/sql/sql.service.ts:67`

所以你可以把这层理解成：
**即使用户已经登录、接口权限也放你进来了，系统还会再看“你在哪个库、你执行什么类型的 SQL、这类 SQL 该不该被拦下”。**

#### 第四层：审批流拦截
一旦 SQL 是 DML / DDL 且不能直执行，就不是“失败结束”，而是被导向审批流。
审批层在：
- `packages/business-dms-approval/src/approvals.service.ts:33`
- `packages/business-dms-approval/src/approvals.service.ts:43`
- `packages/business-dms-approval/src/approvals.service.ts:84`

这里还有几道拦截：

##### 1. 提交审批时先校验参数
没有 `dataSourceId` 或 `sql`，直接 400：
- `packages/business-dms-approval/src/approvals.service.ts:35`

##### 2. 审批时必须处于 pending
不是 `pending` 状态不能审：
- `packages/business-dms-approval/src/approvals.service.ts:50`

##### 3. 提交人不能审自己
- `packages/business-dms-approval/src/approvals.service.ts:53`

##### 4. 执行审批单前必须是 approved / execute_failed
- `packages/business-dms-approval/src/approvals.service.ts:108`

所以就算 SQL 已经进到审批流里，也不是谁都能点执行。

#### 第五层：执行前还有连接层拦截
真正执行前还要再过一层：
- `getPoolForDatabase(...)`
- `getPool(...)`

如果数据源解不出来、池拿不到，也会被挡住：
- `packages/app-dms-server/src/sql/sql.service.ts:123`
- `packages/business-dms-approval/src/approvals.service.ts:125`

#### 第六层：审计并不是拦截，但它是事后控制
执行后还会写审计日志：
- 直执行 SELECT / DML：`packages/app-dms-server/src/sql/sql.service.ts:151`
- 审批执行：`packages/business-dms-approval/src/approvals.service.ts:132`

这个不算前置拦截，但它意味着表操作不是“静默发生”的。

---

### 所以最终答案

**有，而且是多层串联的：**

1. 登录鉴权拦截
2. 路由级权限拦截
3. SQL 参数 / 数据源 / 角色 / SQL 类型校验
4. DML / DDL 审批流拦截
5. 连接池和数据源可用性校验
6. 执行后审计留痕

如果你只关心“改表之前有没有阻断机制”，那答案更短：

**有，DML/DDL 默认会被拦下来，不会直接打到数据库，除非满足测试库直执行条件；否则必须走审批。**

### 2. 深度交互

你这个问题本质上是在问：这个系统到底是“SQL 工具”，还是“受控数据库治理系统”。

如果只是 SQL 工具，通常只有两层：
- 登录
- 执行

但这里明显不是。
这里在“执行前”插了很多决策点：
- 你是谁
- 你在碰哪个库
- 你执行的是什么语句
- 这类语句该不该被强制送审批
- 你是不是审批链上的合法执行者

所以从第一性原理看，这个系统真正的价值不是“能操作表”，而是：

**它把“表操作”从一个技术动作变成了一个治理动作。**

也就是说，系统关注的不是“能不能发 SQL”，而是“谁在什么时候、以什么身份、对什么库、用什么方式发 SQL”。

如果你愿意，我下一步可以继续把这个东西画成一张非常直观的拦截链：

**请求进来 -> authHook -> 权限表 -> sqlParse -> DML/DDL 分流 -> approval -> pool.execute**

你一眼就能看清楚哪里是“放行”，哪里是“拦截”。
