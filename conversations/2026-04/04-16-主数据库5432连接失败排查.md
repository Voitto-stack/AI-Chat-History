### 1. 直接执行
结论先说：这不是 Redis 的问题，也不是 Monitor 库的问题。是 **Minerva 主库 `DATABASE_URL` 指向的 PostgreSQL 在 `localhost:5432` 不可达**，所以 Prisma 主库初始化报了 `P1001`。与此同时，`MONITOR_DATABASE_URL` 指向的另一个库是通的，所以你才会看到 Monitor 数据库查询成功。

证据在这几处：

- 主库启动探测代码在 `apps/minerva-server/src/index.ts:78-97`
  这里直接执行了：
  - `await prisma.$connect()`
  - `await prisma.$queryRaw\`SELECT NOW()\``
  任何一步连不上都会进入 catch，然后打印你现在看到的这段 `[主数据库] ❌ 连接失败`。

- 主库 PrismaClient 用的是 `DATABASE_URL`，在 `apps/minerva-server/src/utils/prisma.ts:4-18, 32-36`
  这里会读取 `process.env.DATABASE_URL`，并打印主机、端口、数据库名。你现在的报错文字已经说明它最终解析到的是 `localhost:5432`。

- Monitor 库是另一套 PrismaClient，在 `apps/minerva-server/src/utils/monitorPrisma.ts:4-18, 33-45`
  它读取的是 `MONITOR_DATABASE_URL`，并且模块加载后立即 `$connect()` 再跑一条查询，所以日志里才会出现：
  - `✅ Monitor 数据库连接成功`
  - `db":"monitor_test"`

也就是说现在实际状态是：
- `DATABASE_URL` → 指向 `localhost:5432` → 连不上
- `MONITOR_DATABASE_URL` → 指向另一个可用库 → 能连上
- Redis 两个地址 `6378/6379` → 都正常

`P1001` 在 Prisma 里就是很直接的含义：**数据库地址能解析，但目标端口上没有可连接的 PostgreSQL 服务**。常见原因就四种：

1. 你本机 5432 根本没启动 Postgres
2. `.env.local` 里的 `DATABASE_URL` 配错了，误写成了本地 `localhost:5432`
3. 你主库其实跑在 Docker / 别的 host / 别的端口，但环境变量还没改
4. 本机确实有 5432，但服务没监听 TCP，或者被别的进程占了

还有一个关键点：服务虽然报错，但**没有退出**。这是 `apps/minerva-server/src/index.ts:91-97` 故意这么写的——主库失败只记日志，服务继续启动。所以你会看到 `Minerva Server 已启动`，但凡是依赖主库 Prisma 的接口，后面运行时还是会继续报错。

### 2. 深度交互
你现在最该优先确认的不是代码，而是**环境分层是不是已经混了**。

因为这个服务现在明显同时依赖两套数据库：
- 一套是主业务库 `DATABASE_URL`
- 一套是 monitor 库 `MONITOR_DATABASE_URL`

而 monitor 已经连到 `monitor_test` 了，说明你的 `.env` 至少部分是有效的。那主库单独失败，通常不是“服务整体没配好”，而是 **`DATABASE_URL` 这一项单独指错了**，或者你本地只起了 monitor/test 库，没起业务主库。

如果从目标倒推，你真正要确认的是：**本地开发到底应不应该依赖一个跑在 `localhost:5432` 的主业务 Postgres**。如果答案是“应该”，那就是服务没起；如果答案是“不应该”，那就是环境变量配置错了。
