### 1. 直接执行

我直接用“代码证明链”回答你，不靠概念。

你说的那一套，如果成立，代码里必须同时满足这 5 个条件：

1. **外部数据库连接信息不是散落在业务代码里，而是被 DMS 集中登记**
2. **真正连库前，必须先经过 DMS 自己的数据源解析 / 解密**
3. **真正执行 SQL 时，不是每个模块自己 new connection，而是统一走同一个池获取入口**
4. **直执行和审批执行都复用这套入口**
5. **业务模块不直接依赖 mysql2 / pg，而是依赖 DMS 抽象层**

这个项目 5 条全中，所以我才说它就是你讲的“自己的数据库代理 / 调度层”。

#### 第一层：数据源先被 DMS 自己管理
看 `packages/business-dms-datasource/src/datasources.model.ts:3`
这里所有数据源都落在 DMS 自己的 app DB（Postgres）里，通过 `getPgDb()` 读写：
- `listDataSources()`：`packages/business-dms-datasource/src/datasources.model.ts:3`
- `getDataSourceRow()`：`packages/business-dms-datasource/src/datasources.model.ts:11`
- `insertDataSource()`：`packages/business-dms-datasource/src/datasources.model.ts:15`
- `updateDataSource()`：`packages/business-dms-datasource/src/datasources.model.ts:45`

这说明“外部数据库是谁、地址是什么、用户名是什么、池大小是多少”不是散落在调用代码，而是先被 DMS 系统托管。

#### 第二层：真正连外部库前，先经过 DMS 解密和组装
看 `packages/business-dms-datasource/src/datasources.service.ts:88`
`getDataSourceWithPassword(id, encryptionKey)` 会：
- 从 DMS 自己的 `data_sources` 记录里取原始行：`packages/business-dms-datasource/src/datasources.service.ts:92`
- 解密 `password_encrypted`：`packages/business-dms-datasource/src/datasources.service.ts:101`
- 返回完整可连接的数据源配置对象

这一步非常关键。
它证明不是业务代码自己拿 host/password 去连，而是必须先穿过 DMS 的“凭据托管层”。

#### 第三层：真正对外部库的连接入口被统一收口到 `getPool()`
看 `packages/business-dms-datasource/src/datasources.service.ts:46`

`getPool(ds)` 就是统一的外部数据库访问入口。
它的实现逻辑是：
- 先按 `dataSourceId:database` 算 key：`packages/business-dms-datasource/src/datasources.service.ts:42`
- 如果池已存在，直接返回缓存池：`packages/business-dms-datasource/src/datasources.service.ts:59`
- 否则根据类型创建 MySQL 或 PostgreSQL 连接池：`packages/business-dms-datasource/src/datasources.service.ts:62`
- 然后塞进 `pools` map：`packages/business-dms-datasource/src/datasources.service.ts:83`

这一步就是你说的“代理 / 调度层”的核心证据。
因为它说明：

**整个系统访问外部数据库时，不是到处自己建连接，而是统一调度到一个集中池管理入口。**

如果再看 `getPoolForDatabase(...)`：
- `packages/business-dms-datasource/src/datasources.service.ts:108`

它进一步把“先查数据源 -> 解密 -> 切到目标 database -> 再交给 getPool”这一套流程封装掉了。

所以统一入口实际上有两个层次：
- `getDataSourceWithPassword()`：拿到可用连接信息
- `getPool()` / `getPoolForDatabase()`：拿到真正可执行 SQL 的池

#### 第四层：底层驱动被包在插件层后面，不暴露给业务
真正创建连接池的底层不是业务模块自己写的，而是插件层：

- PostgreSQL：`packages/server-plugin-pg/src/index.ts:27`
  这里内部才会 `new pg.Pool(...)`：`packages/server-plugin-pg/src/index.ts:28`

- MySQL：`packages/server-plugin-mysql/src/index.ts:17`
  这里内部才会 `mysql.createPool(...)`：`packages/server-plugin-mysql/src/index.ts:18`

注意这个依赖方向：
- `app-dms-server/sql.service.ts` 不直接 import `pg` / `mysql2`
- `business-dms-approval/src/approvals.service.ts` 也不直接 import `pg` / `mysql2`
- 它们统一 import 的是 `@heyhru/business-dms-datasource`

这就是抽象层存在的直接证据。
如果没有这层“自己的代理 / 调度层”，那你会看到业务代码到处直接 `new Pool()` 或 `createPool()`；但这里没有。

#### 第五层：直执行 SQL 和审批执行 SQL 都复用同一套入口
这是最强证明。

##### A. 普通 SQL 执行
看 `packages/app-dms-server/src/sql/sql.service.ts:108`

`sqlExecute(...)` 在真正执行 SQL 前，会先拿 pool：
- 指定 database 时走 `getPoolForDatabase(...)`：`packages/app-dms-server/src/sql/sql.service.ts:117`
- 否则先解数据源，再走 `getPool(ds)`：`packages/app-dms-server/src/sql/sql.service.ts:119`

拿到 pool 后，真正执行统一是：
- `pool.execute(...)`：`packages/app-dms-server/src/sql/sql.service.ts:138`

##### B. 审批后执行 SQL
看 `packages/business-dms-approval/src/approvals.service.ts:106`

`doExecuteApproval(...)` 在真正执行审批单 SQL 前，拿 pool 的逻辑几乎是同一套：
- 有 `dbName` 时走 `getPoolForDatabase(...)`：`packages/business-dms-approval/src/approvals.service.ts:118`
- 否则先解数据源，再走 `getPool(...)`：`packages/business-dms-approval/src/approvals.service.ts:120`

真正执行同样是：
- `pool.execute(approval["sql_text"])`：`packages/business-dms-approval/src/approvals.service.ts:127`

这意味着：

**无论是“直接查库”还是“审批后改库”，都没有绕开 DMS 的统一连接入口。**

这点是决定性的。
如果只有 SQL editor 走这层，而审批执行绕开它，那就不能叫统一接管；但这里两个路径都复用了同一入口，所以可以成立。

### 最后的精确定义

所以我给它的准确定义不是“数据库代理服务器”，而是：

**应用内嵌式的数据库访问代理 / 调度层**

为什么不是传统 DB proxy？
因为它：
- 不是独立进程
- 不转发 TCP/MySQL/Postgres 协议
- 不在网络层拦截连接

为什么仍然是你说的那一套？
因为它在**应用层**完整接管了：
- 外部数据库元信息
- 凭据
- 连接池创建与复用
- 业务执行入口
- 审批与审计工作流

这已经满足“统一接管外部数据库访问”的本质定义了，只是不是网络层代理，而是**应用层代理 / 调度**。

### 2. 深度交互

你这个问题问得很准，因为它逼着我们区分两个经常被混淆的概念：

1. **网络层 DB Proxy**
   - 像 PgBouncer / ProxySQL / MySQL Router
   - 独立服务，拦截数据库协议

2. **应用层 DB Access Orchestrator**
   - 不是拦协议
   - 而是在应用内部统一调度数据库访问

这个仓库明显是第 2 种。

如果从第一性原理看，“代理 / 调度层”的本质不是它跑在哪一层，而是：

**外部数据库访问权是不是先被一个中心控制点拿走，再由这个控制点决定如何连接、如何执行、如何治理。**

这套代码里，那个中心控制点就是 `business-dms-datasource`。
它不是旁支工具包，而是整个 DMS 外部数据库访问的闸门。
