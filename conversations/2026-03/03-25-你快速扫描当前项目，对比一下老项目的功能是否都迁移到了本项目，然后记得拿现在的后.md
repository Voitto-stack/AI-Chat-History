## 更正结论

### 我之前报告的两个"Bug"实际上不是问题

新后端排序是**两层架构**：

```
SQL 层排序（queryCallOrders/queryInsOrders）
       ↓ 两张表各自返回数据
JS 层重新排序（mergeAndSortCallData）  ← 这才是最终排序
```

我之前说的 SQL 层 `MAX(uco.create_at)` 和 `female_user_id DESC` 问题，**不影响最终排序**，因为 JS 层 `mergeAndSortCallData` 会把两张表数据合并后重新排序。

JS 层排序逻辑（L503-549）：
- **TimeRange 模式**：`maxCreateTime DESC → femaleUserId ASC → createTime DESC` → **完全对齐**老后端 L170-173
- **Guild 模式**：`maxJoinTime DESC (NULLS LAST) → femaleUserId ASC → createTime DESC` → **逻辑也写对了**，对齐老后端 L79-82

### 但有一个真正的问题：`_guildJoinTime` 数据从未填充

从 git 历史看：

- **commit 668d703**：首次添加了 `_guildJoinTime` 字段、`CallDataRowInternal` 接口、`mergeAndSortCallData` 排序逻辑
- **commit 0ec3d6c**：把 `_guildJoinTime: number` 改成 `_guildJoinTime?: number`，加了 `?? 0` 容错

**但从头到尾**，`queryCallOrders` 和 `queryInsOrders` 的 SQL 都**没有 SELECT `ucm.created_at`**，rows.map 也**没有映射 `_guildJoinTime`**。

对比老后端 SQL（CallDataDBUtils.java L64）明确有：
```sql
EXTRACT(EPOCH FROM ucm.created_at) * 1000 as join_guild_time,
```

新后端 SQL（call-feed.ts L609-648）完全没有这个字段。

**结果**：
- `_guildJoinTime` 永远是 `undefined → 0`
- `maxJoinTimeByUser` 所有用户的值都是 0
- Guild 排序退化为 `femaleUserId ASC, createTime DESC`（丢失了"加入公会时间"这个排序维度）

### 总结

| 问题 | 状态 |
|-----|------|
| SQL 层排序字段错 | **不是 Bug** — JS 层会覆盖 |
| SQL 层 DESC/ASC 错 | **不是 Bug** — JS 层会覆盖 |
| JS 层 TimeRange 排序 | **正确** |
| JS 层 Guild 排序逻辑 | **正确**（但缺数据） |
| `_guildJoinTime` 从未赋值 | **是 Bug** — SQL 和映射都缺 |

需要修复的是在 `queryCallOrders` 和 `queryInsOrders` 的 SQL 中加上 `COALESCE(EXTRACT(EPOCH FROM ucm.created_at) * 1000, 0) AS join_guild_time`，并在返回映射中添加 `_guildJoinTime: Number(r.join_guild_time)`。
