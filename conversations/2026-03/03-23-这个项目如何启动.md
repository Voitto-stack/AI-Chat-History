## 1. 直接执行

### 项目启动方式

这是一个基于 **Gradle + Vert.x** 的微服务项目，有两种启动方式：

#### 方式一：本地 IDEA 开发启动

1. **前置依赖**：首先需要在 [presence-server-core](https://gitee.com/presence-dev/presence-server-core) 项目执行：
   ```bash
   ./gradlew clean publishToMavenLocal
   ```

2. **构建项目**：
   ```bash
   ./gradlew clean build
   ```

3. **配置 IDEA 环境变量**：
   ```
   ENV=archat_local
   ```

4. **运行**：在 IDEA 中运行 `com.pserver.grpc.Application` 类

#### 方式二：Docker Compose 启动

**开发环境**：
```bash
docker-compose -f docker-compose.dev.yml up -d
```

**生产环境**：
```bash
docker-compose -f docker-compose.prod.yml up -d
```

Docker Compose 会启动多个微服务容器（user-service、post-service、messaging-service 等），每个服务通过 `SERVICE_CLASS` 环境变量区分。

## 📊 数据结构关系

**关键发现**：
- `post_like_count` 表：存储每个帖子的**总点赞数**和**按性别分组的点赞数**
- `post_interact_count` 表：通过 `interact_type` 字段区分不同类型的互动：
  - `MALE_LIKE` / `FEMALE_LIKE` / `NON_BINARY_LIKE` (点赞)
  - `MALE_COMMENT` / `FEMALE_COMMENT` / `NON_BINARY_COMMENT` (评论)

**理论同步关系**：
```sql
post_like_count.male_count = post_interact_count[interact_type='MALE_LIKE'].count
post_like_count.female_count = post_interact_count[interact_type='FEMALE_LIKE'].count  
post_like_count.nonbinary_count = post_interact_count[interact_type='NON_BINARY_LIKE'].count
post_like_count.count = male_count + female_count + nonbinary_count
```

## 💡 我的分析

### **为什么需要两张表？**

这可能是一个**逐步重构**的过程：
1. **旧设计** (`post_like_count`)：只记录点赞，按性别分组
2. **新设计** (`post_interact_count`)：更通用的互动计数表，支持点赞、评论等多种类型

### **Koa 重构的影响**

如果 Koa 项目正在简化架构，可能会：
- 废弃 `post_like_count`，统一使用 `post_interact_count`
- 或者反过来，废弃 `post_interact_count`，只保留简单的 `post_like_count`

**建议**：先运行检查脚本，看数据同步情况，然后决定：
1. 如果同步良好 → 可以废弃其中一张表
2. 如果严重不同步 → 需要先修复数据再重构
